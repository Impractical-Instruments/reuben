Sample-decode loading / progress UX — option comparison

Five candidate presentations of the same simulated load, side by side, so a human can react and pick. Drive them with the controls, then compare how each option handles a slow connection, a fetch 404, and a decode error. This picks no winner — it makes the options contrastable. Timing is faked; the real path decodes WAV in Rust/WASM at stage_resource (see the notes at the bottom and README).
Phases (mirroring the real path):
fetching — fetch(assetBase + key) per miss decoding — hound WAV decode in stage_resource ready — worklet construct returned, audio can play error — fetch or decode failed (non-fatal in core, fatal to this load in JS)

*drumkit is a hypothetical multi-sample rig — no shipping Toy carries samples today (this path is latent), so per-sample vs per-rig progress only becomes visible on a rig like this. The 1-sample rigs show why the distinction may not be worth building yet.
