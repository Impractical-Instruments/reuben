# V1.4 Good-button UX layer — human testing

A hands-on walkthrough of the V1.4 deliverable ([ADR-0018](adr/0018-control-surface-generation.md)):
the **`control-surface` skill** turns an Instrument into a **Hexler TouchOSC** layout (`.tosc`)
you actually touch on a phone or tablet, sending OSC straight to reuben's node addresses.

This is the surface a human plays — driven over OSC first, exactly as the roadmap's V1.4 calls
for, with **no native GUI commitment**. The TouchOSC layout *is* the "stand-in" surface that
proves the control-surface API. (The reactive auto-UI / native-or-web app is explicitly deferred
to *Later (post-v1)* — this generator de-risks it by getting us playing first.)

If `cargo test` is what *proves* the engine works, and `docs/v1.2-playable-surface-testing.md` is
how you *feel* the control operators, this is how you *hold* an instrument in your hands.

## Setup

1. **Run an instrument** (opens your default audio device, listens on UDP `0.0.0.0:9000`):

   ```sh
   cargo run -p reuben-native --bin reuben -- play instruments/good-button.json
   ```

2. **Install Hexler TouchOSC** on a phone/tablet (or the desktop editor) — <https://hexler.net/touchosc>.

3. **Point TouchOSC at reuben.** The generated `.tosc` carries the layout and the OSC *messages*
   (addresses + value ranges); it does **not** bake in a host. Set the destination in TouchOSC's
   **connections** settings: the IP of the machine running reuben, port **9000**, send enabled.
   (The skill's `--host`/`--port` flags only print a reminder of what to enter — they do not alter
   the file. The tablet and the reuben machine must share a network.)

> Tip: external OSC is block-quantized by design (it applies at the next block boundary) — that
> is honest, not a bug. Sample-accurate timing is the Clock's job, not a UDP datagram's.

The skill's directory is `.claude/skills/control-surface/`; its script is `gen_surface.py`. Run
everything below from the repo root.

## 1. The Good Button surface — `good-button.json`

The canonical Good Button: one **Brightness** knob fanned to filter cutoff *and* resonance
([ADR-0017](adr/0017-playable-surface-and-control-domain.md)). The instrument is already
annotated with `control` blocks, so the surface is a single `emit`.

**See what the surface will contain** (read-only):

```sh
python3 .claude/skills/control-surface/gen_surface.py infer instruments/good-button.json
```

`infer` prints the high-confidence **Good Button** (`/brightness`, `[0,1]`, rests at `0.5`) and
every node param with its resolved address + metadata, so you can sanity-check ranges before
emitting.

**Emit the surface:**

```sh
python3 .claude/skills/control-surface/gen_surface.py emit instruments/good-button.json
# wrote control-surfaces/good-button.tosc — 6 control(s)
```

Six controls, from the instrument's stored `control` blocks: **Play C** (a `note-toggle` that
fires `/voicer/notes [60, gate]`), the four **envelope** faders (Attack/Decay/Sustain/Release,
each `0.001–5 s`, Sustain `0–1`), and the **Brightness** fader.

**Open `control-surfaces/good-button.tosc` in TouchOSC and play:**

- Tap **Play C** — it latches the note on; tap again for note-off (a toggle reuses one constant
  note so note-off matches; TouchOSC can't share a value between a slider and a gate without
  scripting).
- With the note held, sweep **Brightness** `0 → 1`: the tone opens from dull/closed to a bright,
  resonant peak — *one* finger moving *two* targets over their own ranges. No knob position
  sounds bad: that is the Good Button (principle) made touchable.
- Move **Attack/Decay/Sustain/Release** and re-tap Play C to reshape the envelope.

**What "passing" looks like:** the brightness knob audibly opens cutoff **and** resonance
together and smoothly; the ADSR faders reshape the note; Play C sounds and silences cleanly.

## 2. A second surface — `djfilter-demo.json`

A clock-driven sequence through the **DJ filter** operator, with a single bipolar **Filter** knob
(LP below center, HP above). Also pre-annotated:

```sh
python3 .claude/skills/control-surface/gen_surface.py emit instruments/djfilter-demo.json
# wrote control-surfaces/djfilter-demo.tosc
```

Three controls: **Filter** (`/filter_knob`, `[-1, 1]`, rests at North = `0`), **Resonance**
(`/djfilter/resonance`), and **Tempo** (`/clock/tempo`, BPM).

**Play it:**

```sh
cargo run -p reuben-native --bin reuben -- play instruments/djfilter-demo.json
```

Open the surface and ride the **Filter** knob while the sequence runs: center is open, sweep down
for a low-pass close, up for a high-pass thin-out — the classic DJ filter gesture, on a tablet.
**Resonance** sharpens the corner; **Tempo** drives the sequence speed.

## 3. Annotate a fresh instrument — the inference + write-back pass

The first two instruments already had `control` blocks. Most instruments don't. On a fresh one
the skill **infers candidates, proposes a tight set, and writes `control` blocks back into the
JSON** — the step that earns it over hand-authoring. Here is that pass on `sequence.json` (an
8-step sequencer with no controls yet).

**Infer** — note there are zero Good Buttons, but several genuinely *musical* params:

```sh
python3 .claude/skills/control-surface/gen_surface.py infer instruments/sequence.json
```

```
/clock/tempo        1–999 BPM     (default 120)
/sequencer/length   1–8 steps     (default 8)
/sequencer/step1..8 -1–24 degree  (scale degree per step; -1 = rest)
/voicer/voices      1–32          (default 8)
```

**Curate.** The skill proposes the play-worthy controls and skips structural/internal params.
For a step sequencer that is **Tempo + the eight step pitches** (compose the melody live from the
tablet), optionally **Length** and **Voices**. It then writes `control` blocks into
`instruments/sequence.json`, e.g.:

```json
{ "type": "clock", "address": "/clock",
  "params": { "tempo": 120 },
  "control": { "label": "Tempo", "param": "tempo" } }
```

```json
{ "type": "sequencer", "address": "/sequencer",
  "params": { "length": 8, "step1": 0, "step2": 2, ... },
  "control": [
    { "label": "Step 1", "param": "step1" },
    { "label": "Step 2", "param": "step2" },
    ...
  ] }
```

`label` is required (inference can't supply a human name); `unit`/`widget`/`min`/`max`/`default`
are optional overrides, otherwise taken from each param's schema metadata. The blocks are
**engine-ignored passthrough** — they round-trip through load → re-serialize and change nothing
about how the instrument sounds.

**Emit** the surface from the now-annotated file:

```sh
python3 .claude/skills/control-surface/gen_surface.py emit instruments/sequence.json
```

Open it and **set each step's pitch** live while the sequence plays — every step is a scale
degree, so you stay in key by construction. `infer` on a second run is now deterministic from the
stored blocks; the annotation pass is one-time.

## What "passing" looks like

- **`good-button`**: one knob opens cutoff **and** resonance together, smoothly; ADSR + Play C
  respond.
- **`djfilter-demo`**: the bipolar Filter knob sweeps LP↓ / HP↑ around center; Resonance and
  Tempo respond.
- **`sequence`** (after the inference pass): the file gains `control` blocks, the emitted surface
  exposes Tempo + 8 step pitches, and editing a step changes the live melody — all in key.
- The skill's own tests stay green: `python3 -m pytest .claude/skills/control-surface/` (17 tests
  covering `infer`, `emit`, the binding table, and the `.tosc` structure).

The surface is disposable (ADR-0018): the instrument JSON is the source of truth; regenerate the
`.tosc` whenever the instrument changes.
