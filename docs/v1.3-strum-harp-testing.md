# V1.3 Strum harp — human testing

A hands-on walkthrough of **Toy 3 of 3** ([ADR-0022](adr/0022-the-toys.md) §3): a drag-to-strum
harp you play from a **Hexler TouchOSC** surface. One big fader is the strum bar — drag it and the
new [`strum`](../crates/reuben-core/src/operators/strum.rs) operator plucks a note each time the
position crosses a string boundary, in key, like a thumb across a harp.

This is the continuous-gesture Toy: where the groove box loops and the chord player taps, the harp
is one sweeping drag. It reuses the fader the generator already draws — no new widget type — so the
whole surface is `strum bar + brightness + key + range`.

If `cargo test` is what *proves* the engine works, this is how you *hold* the harp in your hands.

## Setup

1. **Run the instrument** (opens your default audio device, listens on UDP `0.0.0.0:9000`):

   ```sh
   cargo run -p reuben-native --bin reuben -- play instruments/strum-harp.json
   ```

2. **Install Hexler TouchOSC** on a phone/tablet (or the desktop editor) — <https://hexler.net/touchosc>.

3. **Point TouchOSC at reuben.** Open `control-surfaces/strum-harp.tosc`. Set the destination in
   TouchOSC's **connections** settings: the IP of the machine running reuben, port **9000**, send
   enabled. (The `.tosc` carries the layout + OSC addresses, not a host; the tablet and the reuben
   machine must share a network.)

> Tip: external OSC is block-quantized by design — it applies at the next block boundary. A drag
> streams many position messages, so the harp still feels continuous; the strum op holds its band
> state across blocks, so a crossing that straddles a block boundary plucks exactly once.

Run everything below from the repo root.

## The surface — `strum-harp.json`

Four controls, from the instrument's stored `control` blocks
([ADR-0018](adr/0018-control-surface-generation.md)):

| Control | OSC address | Range | What it does |
|---|---|---|---|
| **Strum** (fader) | `/strum/position` | `0..1`, rests at `0.0` | the strum bar — drag to pluck strings |
| **Range** (knob) | `/strum/octaves` | `1..4` oct, default `1` | how many octaves the strings span |
| **Brightness** (fader) | `/brightness` | `0..1`, rests at `0.6` | opens filter cutoff **and** resonance (a Good Button) |
| **Key** (selector) | `/harmony/root` | MIDI `48..60` (C3..C4), default `60` | transposes the whole harp, live and always in key |

**See what the surface contains** (read-only):

```sh
python3 .claude/skills/control-surface/gen_surface.py infer instruments/strum-harp.json
```

`infer` prints the one **Good Button** (`/brightness`) and the direct params (`/strum/position`,
`/strum/octaves`); the curated set written into the JSON is the four controls above.

**(Re)emit the surface** if you edit the instrument:

```sh
python3 .claude/skills/control-surface/gen_surface.py emit instruments/strum-harp.json
# wrote control-surfaces/strum-harp.tosc — 4 control(s), send to localhost:9000
```

## Play it

Open `control-surfaces/strum-harp.tosc` in TouchOSC and:

- **Drag the Strum fader slowly from bottom to top.** You should hear an **ascending glissando** —
  eight plucked notes climbing one diatonic octave (C major by default), one per string boundary
  the fader crosses. Each note is a *pluck*: it rings out on the percussive envelope and decays on
  its own, so a sweep leaves a shimmering trail of overlapping notes (the harp is polyphonic, 8
  voices).
- **Drag it back down.** The same strings pluck in **descending** order — both directions strum.
- **Flick it fast across the whole range.** Every string between your start and end plucks in
  sequence, fast — a thumb-across-the-strings glissando.
- **Hold it still inside one band.** Silence: with no boundary crossed, nothing plucks. The harp
  only speaks on movement.

Then shape the tone:

- **Brightness `0 → 1`:** the plucks open from dull and closed to bright and resonant — one finger
  moving filter cutoff *and* resonance over their own ranges (the Good Button).
- **Range `1 → 4`:** widens the span the eight strings cover. At `1` a sweep is one octave; at `2`
  the same drag covers two octaves (bigger leaps between strings); higher still for a wide, sparse
  harp.
- **Key:** step it down from `C` and **re-drag the strum** — the whole glissando transposes to the
  new key, still perfectly in scale. This is reuben's signature: the strings are scale *degrees*
  resolved live through the tonal context, never fixed pitches.

## What "passing" looks like

- A slow up-drag plays a clean **ascending** in-key glissando; a down-drag plays it **descending**.
- A motionless fader is silent; movement across a boundary plucks exactly one note per string.
- **Brightness** opens cutoff and resonance together, smoothly; **Range** widens the sweep;
  **Key** transposes the harp live with no edit and no out-of-key note.
- The op's own oracle stays green: `cargo test -p reuben-core operators::strum` (9 tests covering
  ascending/descending sweeps, no-crossing silence, fast-drag glissando, the pluck note-off pairing,
  the octave span, cross-block continuity, and spawn reset).

The surface is disposable (ADR-0018): `instruments/strum-harp.json` is the source of truth;
regenerate the `.tosc` whenever the instrument changes.

> **Honest audible status:** the graph validates, the strum op meets its written oracle, and the
> note/degree/harmony wiring matches the proven `scale-demo` path — but the *timbre* (triangle
> osc + percussive envelope) and the **pluck ring length** (`PLUCK_SAMPLES` ≈ 30 ms note-on before
> the paired note-off) are tuned by ear, not by test. If the plucks choke or ring too long, adjust
> the envelope `decay`/`release` in the instrument or the constant in `strum.rs`.
