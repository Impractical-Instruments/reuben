# V1.2 Playable surface — human testing

A hands-on walkthrough of the V1.2 control surface ([ADR-0017](adr/0017-playable-surface-and-control-domain.md)):
**Good Buttons**, the **math-operator family** (`map`, `add`, `mul`, `differentiate`,
`integrate`), the **Message→Signal converter** (`m2s`), and the **one-port-one-type** filter
(cutoff/resonance are now Signal inputs you can sweep).

Everything is driven over OSC — exactly how a Toy or a UI will drive it later. If the
automated tests are what *prove* it works (`cargo test`), this is how you *feel* it work.

## Setup

1. **Run an instrument** (opens your default audio device, listens on UDP `0.0.0.0:9000`):

   ```sh
   cargo run -p reuben-native --bin reuben -- instruments/good-button.json
   ```

2. **Pick an OSC sender.** Any will do; examples below give two forms.

   - **`oscsend`** (from `liblo-tools` — `apt install liblo-tools`):
     ```sh
     oscsend localhost 9000 /voicer/note ff 69.0 1.0
     ```
     The `ff` is the type tag (two floats); `f` = one float.

   - **`python-osc`** (`pip install python-osc`):
     ```python
     from pythonosc.udp_client import SimpleUDPClient
     c = SimpleUDPClient("127.0.0.1", 9000)
     c.send_message("/voicer/note", [69.0, 1.0])
     ```

   In the recipes below each step is written as `ADDRESS  args` — translate to your sender.

> Tip: external OSC is block-quantized by design (it applies at the next block boundary) —
> that is honest, not a bug. Sample-accurate timing is the Clock's job, not a UDP datagram's.

## 1. The Good Button — `good-button.json`

One **brightness** knob fanned to filter **cutoff *and* resonance**, each over its own range,
each smoothed into the audio rate. The whole thing is composition: a public identity `map`
→ two ranged `map`s → two `m2s` converters → the filter's Signal `cutoff`/`resonance` inputs.
No new instrument-format machinery.

Hold a note, then sweep the one knob:

```
/voicer/note   ff 57.0 1.0     # note-on, A3 (a saw — lots of harmonics to filter)
/brightness    f  0.0          # darkest: cutoff ~400 Hz, low resonance
/brightness    f  0.3
/brightness    f  0.6
/brightness    f  1.0          # brightest: cutoff ~12 kHz, high resonance
/voicer/note   ff 57.0 0.0     # note-off
```

**What to listen for:** at `0.0` the tone is dull and closed; as `brightness` rises it opens
up and the resonance adds a vocal "peak" — *one* gesture moving *two* targets over *different*
ranges. The motion is smooth (the `m2s` converters are in `smooth` mode), not stepped.

**Why it's a Good Button:** there is no value of `/brightness` in `[0,1]` that sounds bad —
the ranges and the exponential cutoff curve are baked into the `map` params. Energy in →
musical result out.

Peek under the hood (every internal node is live over OSC too):

```
/cv_cutoff/default  f 6000     # change the resting cutoff before the first knob move
/map_res/out_max    f 0.6      # re-range the resonance fan target
```

## 2. Base + modulation with `add` — `auto-filter.json`

```sh
cargo run -p reuben-native --bin reuben -- instruments/auto-filter.json
```

A Signal **`add`** sums a base-cutoff CV (an `m2s` converter, default 2000 Hz) with an **LFO**
wobble, feeding the filter's Signal cutoff. This is the ADR's "base-plus-modulation built
*explicitly* with an `add` operator" — no hidden param+CV combining.

```
/voicer/note  ff 45.0 1.0      # hold a low note
/lfo/rate     f  2.0           # wobble speed (Hz)
/lfo/depth    f  1500.0        # wobble width (Hz of cutoff swing)
/cutoff       f  800.0         # move the base the wobble rides on
/cutoff       f  3000.0
/lfo/depth    f  0.0           # depth 0 -> static cutoff: the wobble stops
```

**What to listen for:** a classic auto-wah. `/cutoff` slides the whole sweep up/down;
`/lfo/depth` widens/narrows it; `depth 0` freezes it (proving the LFO→`add`→cutoff path is
what's moving). The filter is resonant, so the moving peak sings.

## 3. Poke the operators directly (optional)

The math operators are ordinary nodes — you can build a one-liner rig or just send to the
demo rigs' internal nodes. A few things worth confirming by hand:

- **`map` is an affine remap, clamped.** In `good-button.json`, `/map_cutoff` is
  `[0,1]→[400,12000]` exponential. Sending `/brightness 2.0` (out of range) still produces a
  *clamped* result, never a runaway — the `map` clamps to its input range.
- **`m2s` modes.** Change `/cv_cutoff/mode` (0 = snap, 1 = slew, 2 = smooth, 3 = glide) and
  re-sweep `/brightness`:
  ```
  /cv_cutoff/mode f 0    # snap: stepped, sample-accurate
  /cv_cutoff/mode f 3    # glide: fixed-time portamento (set the time with /cv_cutoff/time)
  ```
- **Cross-domain wiring is a type error.** Try authoring a rig that wires a Message output
  straight into a Signal input (e.g. `/map_cutoff/out` → `/filter/cutoff` *without* the `m2s`
  converter). Loading fails with `PortKindMismatch` — the converter is the *only* sanctioned
  Message→Signal bridge, and the engine enforces it.

## 4. Regression check (the existing rigs still work)

The one-port-one-type sweep changed the filter's interface (cutoff/resonance became Signal
inputs with the params as unwired defaults). Confirm nothing regressed:

```sh
cargo run -p reuben-native --bin reuben -- instruments/default.json
# /voicer/note ff 69.0 1.0  -> a clean A4; /filter/cutoff f 800 still darkens it
```

A static `/filter/cutoff 800` still works exactly as before — the param is now the Signal
port's unwired default, so unwired use is bit-identical (and covered by
`wired_cutoff_input_matches_equivalent_param` in the filter tests).

## What "passing" looks like

- `good-button`: one knob audibly opens cutoff **and** resonance together, smoothly.
- `auto-filter`: `/lfo/depth` widens an auto-wah; `depth 0` freezes it.
- `default`/`echo`/`reverb`/etc.: unchanged from V1.1.
- `cargo test --workspace` green, including `good_button_brightness_opens_the_filter`,
  `auto_filter_base_plus_lfo_modulation_sounds_and_wobbles`, and the `rt_safe` allocation
  check covering the emit + converter paths.
