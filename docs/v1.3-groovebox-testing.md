# V1.3 Toy 1 — Groove box — human testing

A hands-on walkthrough of the first launch Toy ([ADR-0022](adr/0022-the-toys.md)): a free-running,
step-sequenced **synthesized** drum machine you play from **Hexler TouchOSC** over OSC. Three lanes
— kick, snare, hat — loop forever the instant you hit `play`; toggle steps on a 3×16 grid, ride the
lane volumes and a master-tone Good Button, and the beat reshapes live.

Like the V1.4 surfaces ([docs/v1.4-control-surface-testing.md](v1.4-control-surface-testing.md)),
the `.tosc` carries the layout and the OSC *messages* (addresses + ranges) but **no host** — you set
the destination IP/port in TouchOSC. Nothing is sampled: every drum is built from operators (the
reuben thesis — everything is a graph).

## Setup

1. **Run the groove box** (opens your default audio device, listens on UDP `0.0.0.0:9000`):

   ```sh
   cargo run -p reuben-native --bin reuben -- play instruments/groovebox.json
   ```

   It should start grooving immediately — a four-on-the-floor-ish kick, a backbeat snare, and
   offbeat hats. There is **no transport**: the shared `clock` free-runs, so sound is instant.

2. **Install Hexler TouchOSC** on a phone/tablet (or the desktop editor) — <https://hexler.net/touchosc>.

3. **Point TouchOSC at reuben.** Open `control-surfaces/groovebox.tosc`, then in TouchOSC's
   **connections** settings set the host to the IP of the machine running reuben, port **9000**, send
   enabled. (The tablet and the reuben machine must share a network.)

> External OSC is block-quantized by design — it applies at the next block boundary. Sample-accurate
> timing is the Clock's job; a step toggle landing a block late is honest, not a bug.

Run everything below from the repo root.

## What the instrument is

Per lane (`/kick`, `/snare`, `/hat`):

```
clock(gate, division=4) ─→ sequencer(gate_mode=1, 16 steps) ─→ voicer(1) ─→ drum-synth ─→ mul(lane vol) ─┐
                                                                                                          ├─ add (mix) ─→ master filter (Tone) ─→ trim ─→ out
```

- **One shared `clock`** at **`division`=4** — its beat `gate` pulses **4× per beat**, a 16th-note
  grid. All three sequencers advance on that gate.
- **`sequencer` `gate_mode`=1, 16 steps** — each `stepN` is a **boolean on/off**; every hit emits the
  lane's single `pitch` degree (default 0). This is the new V1.3 step-grid behavior.
- **Drum synths (no samples):**
  - **Kick** = a sine `oscillator` whose frequency is a base (`/kick_base`, 48 Hz) **plus** a fast
    pitch-drop envelope (`/kick_penv` scaling a 220 Hz sweep) — the classic "thump." No noise.
  - **Snare** = `noise` (body) **+** a tonal sine `oscillator` (`/snare_osc`, 185 Hz ring), each
    through its own short envelope, summed.
  - **Hat** = `noise` → **highpass** `filter` (`mode`=1, the new V1.3 filter mode) → a very short
    envelope — a thin metallic tick.
- **Lane volume** = a `map` Good Button → `m2s` CV → `mul` on the lane audio.
- **Master tone** = the `/cutoff` Good Button → `m2s` CV → the master lowpass `filter`'s cutoff. A
  `trim` `mul` after it keeps overlapping hits from clipping.

## The surface — `control-surfaces/groovebox.tosc`

**53 controls**, all written into the instrument's `control` blocks:

- **48 step toggles** — a flat 3×16 grid: `K1..K16` (kick), `S1..S16` (snare), `H1..H16` (hat). Each
  sends `/<lane>/stepN [0..1]`; **top = step on, bottom = step off** (drag/tap to the top of a step's
  travel to enable it, to the bottom to silence it). The flat grid *is* the step-grid idiom (ADR-0022).
- **Tempo** — `/clock/tempo`, BPM.
- **Kick Vol / Snare Vol / Hat Vol** — the three lane-volume Good Buttons (`/kick_vol`, `/snare_vol`,
  `/hat_vol`, each `[0,1]`).
- **Tone** — the master-filter Good Button (`/cutoff`, `[0,1]`, rests at 0.8 → bright).

> **Why faders, not native toggle buttons:** the generator draws a hardware *button* only for the
> `note-toggle` widget, which fires `[note, gate]` to a message *port* — but a sequencer step is a
> *param* (`/kick/step3 [value]`), set from the message's first arg. A note-toggle's first arg is a
> constant, so it can't carry on→off. A `[0,1]` fader is therefore the faithful step control here
> (snap to top/bottom = on/off) without touching the shared generator code (the chord-player Toy owns
> generator changes). A native momentary/latching step button is a clean fast-follow once a
> param-toggle widget lands.

**(Re)generate the surface** any time the instrument changes (it is disposable; the JSON is the
source of truth):

```sh
python3 .claude/skills/control-surface/gen_surface.py emit instruments/groovebox.json
# wrote control-surfaces/groovebox.tosc — 53 control(s)
```

**Inspect what it will contain first** (read-only):

```sh
python3 .claude/skills/control-surface/gen_surface.py infer instruments/groovebox.json
```

## Play it

1. **Listen to the default pattern.** Kick on 1/5/9/13, snare on 5/13, hats on the odd 16ths — a
   basic loop, audible the moment `play` starts.
2. **Build a beat on the grid.** Push **K3** and **K7** to the top to add kicks; pull a hat down to
   thin the pattern. Every change loops in on the next bar.
3. **Move the snare.** Toggle **S3** / **S11** for a different backbeat feel.
4. **Ride the lane volumes.** Pull **Hat Vol** down to tuck the hats under the kick; push **Snare Vol**
   up for a cracking backbeat.
5. **Sweep Tone.** Drag the **Tone** Good Button `0 → 1`: the whole kit goes from dark/muffled to
   bright and open — one finger over the master lowpass cutoff.
6. **Change Tempo.** Slide **Tempo** and the groove tracks immediately.

**What "passing" looks like:**

- The instrument **self-plays** the moment you run it — kick, snare, and hat are each audibly
  distinct (low thump / noisy crack / high tick), nothing clips.
- Toggling a step on the grid **adds or removes that hit** on the next loop, per lane.
- **Tone** opens and closes the whole kit smoothly; the lane volumes balance the three drums;
  **Tempo** changes the speed.

## Regression / gate

The engine work behind this Toy stays green:

```sh
cargo test -p reuben-core          # noise / sequencer gate-mode+16-step / filter mode / clock division oracles
cargo test --workspace
cargo run -p reuben-native --bin reuben -- validate instruments/groovebox.json
```

The new/modified operator contracts:

```sh
cargo run -p reuben-native --bin reuben -- describe noise --json      # white-noise source, out [-1,1]
cargo run -p reuben-native --bin reuben -- describe sequencer --json  # + gate_mode, pitch, step9..16
cargo run -p reuben-native --bin reuben -- describe filter --json     # + mode (0 LP / 1 HP / 2 BP)
cargo run -p reuben-native --bin reuben -- describe clock --json      # + division
```

And the surface generator's own tests:

```sh
cd .claude/skills/control-surface && python3 -m unittest   # 17 tests
```

The surface is disposable (ADR-0018): regenerate the `.tosc` whenever the instrument changes.
