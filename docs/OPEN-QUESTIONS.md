## Major threads (ungrilled)

- **Playable surface / performance input.** *(Largely decided — see [ADR-0017](adr/0017-playable-surface-and-control-domain.md): Message-first control, one-port-one-type, the math-operator family, the M→S converter, and Good Buttons built from composition with no format change.)* Remaining — the **Instrument contract / nesting** sub-thread, where the public boundary becomes a first-class declaration: surface → synthesized `Descriptor` (with metadata inherited from the internal param, overridable); a stable node `id` distinct from `address` (opt-in refactor safety); additive-then-encapsulating surface semantics; the address-rename refactor tool (with id auto-pin); and named gesture Operators (tap-to-chord, drag/strum) once there is UI to drive them. Overlaps the **Format and library** thread (id unification across connections + resource refs).
- **Operator authoring (Rust).** The concrete authoring contract: the Operator trait, the descriptor macro/derive, the process signature (how a single-Lane operator receives input blocks + current param values + event lists and writes output), state ownership. Grounds the AI operator-authoring skill and the MVP's ~5 operators.
- **Audio + n-channel device layer.** *(Partly decided — see [ADR-0026](adr/0026-v1-finish-line-osc-out-and-stereo.md): V1.5 ships **stereo** by keeping the Signal mono and modelling channels as separate edges on an N-wide **logical** master bus (indexed `channel` taps; omitted → broadcast). The **logical→device mapping lives in `audio.rs`** — the home for non-stereo-device policy. **Full** n-channel (a multi-channel Signal touching every operator) stays deferred to Later.)* Remaining: sample-rate and block-size negotiation, the full n-channel model, and xrun/dropout policy (what Render does when a block misses its deadline).
- **Toy design.** *(Decided — see [ADR-0022](adr/0022-the-toys.md): three Toys, one per distinct player gesture — a synthesized multi-track **groove box**, a tap-harmony **chord player**, and a drag **strum/harp** — each one Instrument JSON + generated `.tosc`. Settles the new engine work they force: `sequencer` `gate_mode`/16-step, a `noise` op, `filter` `mode`, `clock` `division`, and the two parked **gesture operators** `chord` (degree-in-arg) and `strum`. Melody-player and meta-effects deferred as overlapping. The `[degree,gate]` button payload is the one `control-surface` generator extension.)*
- **Format and library.** *(Partly decided: sample-data references + the resource-load step landed with [ADR-0016](adr/0016-sample-player-and-resource-store.md) — a top-level `resources` table, a `ResourceResolver` seam, dedup at load.)* Remaining: the referenced-**Instrument** library (resolution, naming, versioning); a configurable sample-root and non-file sources; how formats and saved Instruments/Rigs migrate as the system evolves; and save round-tripping a resource binding (a built `Graph` does not retain the logical id today).
- **Introspection / query API.** *(Decided — see [ADR-0020](adr/0020-introspection-and-patcher-skill.md): a thin CLI over the live registry + loader — `reuben describe [op]` (operator ports/params/resources) and `reuben validate <path>` (the real load + `Plan::instantiate` path, no audio; catches kind-mismatches and cycles), both with `--json`. Grounds and closes the build→check→fix loop for the Patcher skill.)* Remaining: **live-graph query** — listing/inspecting/traversing a *running* rig's state — deferred until a running-engine consumer exists (reuben is OSC-in only today, with no handle to interrogate).

## Smaller parked items

- **Block-slicing granularity** — global vs per-operator/cluster ([ADR-0011](adr/0011-message-delivery-and-timing.md)). A Render-internal optimization; contract is settled either way.
- **Buffer-pool ownership** across cluster boundaries (cache locality, who owns edge buffers).
- **Scheduling policy** — static core-pinning vs dynamic work-stealing for the executor.
- **Cycle delay granularity** — feedback unit delay of one block vs one sample.
- **Engine start/stop** lifecycle details (acquiring the audio device, running the Render thread) — noted as orthogonal to Plans in [ADR-0009](adr/0009-graph-lifecycle.md).
